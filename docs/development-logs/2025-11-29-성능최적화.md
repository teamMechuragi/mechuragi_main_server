## 작업 1: 데이터베이스 연결 풀 설정

### 무엇을 하는가?
Spring Boot의 데이터베이스 연결 풀(HikariCP) 설정을 추가합니다.

### 왜 필요한가?
- **현재 문제:** 기본 10개 연결로 동시에 3-4명만 처리 가능
- **해결:** 30개 연결로 증가하여 동시에 15명 이상 처리 가능
- **영향:** 사용자 10명만 동시 접속해도 연결 풀 고갈 발생 가능성 제거

### 변경 파일
`src/main/resources/application-dev.yml`

### 추가 내용
```yaml
datasource:
  hikari:
    maximum-pool-size: 30        # 최대 연결 수 (10 → 30)
    minimum-idle: 10             # 최소 유지 연결 수
    connection-timeout: 30000    # 연결 대기 시간 (30초)
    idle-timeout: 300000         # 유휴 연결 유지 시간 (5분)
    max-lifetime: 1800000        # 연결 최대 수명 (30분)
```

### 예상 효과
- 동시 처리 용량: 3-4명 → 15명 (약 4배 증가)
- 연결 대기 시간 감소
- 타임아웃 에러 방지

### 추가 비용
**없음** (메모리만 약간 증가)

## 작업 2: Redis 메모리 제한 설정

### 무엇을 하는가?
Redis 서버에 메모리 사용 제한과 eviction 정책을 설정합니다.

### 왜 필요한가?
- **현재 문제:** Redis가 무제한으로 메모리 사용 → 서버 메모리 고갈 가능
- **해결:** 500MB 제한 + LRU 삭제 정책으로 안정성 확보
- **시나리오:** 100만 투표 시 450MB+ 사용 예상

### 변경 파일
`docker-compose.yml`

### 변경 내용
```yaml
redis:
  image: redis:alpine
  command: redis-server --maxmemory 500mb --maxmemory-policy allkeys-lru --appendonly yes
```

**설정 설명:**
- `--maxmemory 500mb`: 최대 500MB까지만 사용
- `--maxmemory-policy allkeys-lru`: 메모리 초과 시 가장 오래된 키부터 삭제
- `--appendonly yes`: 데이터 영속성 유지 (재시작 시 복구)

### 예상 효과
- 메모리 무제한 증가 방지
- 서버 다운 위험 제거
- 오래된 투표 데이터 자동 정리

### 추가 비용
**없음**

## 작업 3: 스케줄러 중복 실행 방지 (ShedLock)

### 무엇을 하는가?
블루그린 배포 시 두 인스턴스에서 스케줄러가 동시 실행되는 문제를 방지합니다.

### 왜 필요한가?
**현재 문제:**
```
배포 프로세스:
T0:00 → Green 컨테이너 시작 (스케줄러 시작)
T0:10 → 헬스체크 대기 (최대 60초)
T0:40 → 헬스체크 통과, Nginx 전환
T0:50 → Blue 컨테이너 종료
       ↑
    이 50초 동안 Blue와 Green 스케줄러 둘 다 실행!
```

**결과:** 투표 만료 알림이 2번 발송됨

### 변경 파일
1. `pom.xml` - ShedLock 의존성 추가
2. 새 파일: `src/main/java/com/mechuragi/mechuragi_server/global/config/ShedLockConfig.java`
3. `src/main/java/com/mechuragi/mechuragi_server/domain/vote/scheduler/VoteNotificationScheduler.java`

### 작업 3-1: 의존성 추가

**pom.xml에 추가:**
```xml
<dependency>
    <groupId>net.javacrumbs.shedlock</groupId>
    <artifactId>shedlock-spring</artifactId>
    <version>5.10.0</version>
</dependency>
<dependency>
    <groupId>net.javacrumbs.shedlock</groupId>
    <artifactId>shedlock-provider-jdbc-template</artifactId>
    <version>5.10.0</version>
</dependency>
```

### 작업 3-2: ShedLock 설정 클래스 생성

**ShedLockConfig.java 생성:**
```java
@Configuration
@EnableSchedulerLock(defaultLockAtMostFor = "58s")
public class ShedLockConfig {
    @Bean
    public LockProvider lockProvider(DataSource dataSource) {
        return new JdbcTemplateLockProvider(
            JdbcTemplateLockProvider.Configuration.builder()
                .withJdbcTemplate(new JdbcTemplate(dataSource))
                .usingDbTime()
                .build()
        );
    }
}
```

### 작업 3-3: 스케줄러에 어노테이션 추가

**VoteNotificationScheduler.java 수정:**
```java
@Scheduled(cron = "0 * * * * *")
@SchedulerLock(
    name = "notifyVotesEndingSoon",
    lockAtMostFor = "58s",
    lockAtLeastFor = "30s"
)
public void notifyVotesEndingSoon() { ... }

@Scheduled(cron = "0 * * * * *")
@SchedulerLock(
    name = "completeExpiredVotes",
    lockAtMostFor = "58s",
    lockAtLeastFor = "30s"
)
public void completeExpiredVotes() { ... }
```

**어노테이션 설명:**
- `name`: 락 식별자 (유니크해야 함)
- `lockAtMostFor`: 최대 58초 동안만 락 유지 (스케줄 간격보다 짧게)
- `lockAtLeastFor`: 최소 30초는 다른 인스턴스가 대기

### 작업 원리
```
Blue 인스턴스: 스케줄러 실행 → DB에 락 획득 → 작업 수행
Green 인스턴스: 스케줄러 실행 → DB 락 확인 (이미 있음) → 대기
                               → 30초 후에도 락 있으면 → 스킵
```

### 예상 효과
- 배포 시 중복 알림 발송 방지
- 클러스터 환경에서 안정적 스케줄러 운영
- 데이터 정합성 유지

### DB 테이블 자동 생성
ShedLock이 첫 실행 시 자동으로 `shedlock` 테이블 생성:
```sql
CREATE TABLE shedlock (
    name VARCHAR(64) NOT NULL,
    lock_until TIMESTAMP NOT NULL,
    locked_at TIMESTAMP NOT NULL,
    locked_by VARCHAR(255) NOT NULL,
    PRIMARY KEY (name)
);
```

### 추가 비용
**없음** (DB 테이블 1개 추가, 용량 무시할 수준)

## 작업 4: Flyway 마이그레이션 도구 추가

### 무엇을 하는가?
DB 스키마 변경을 자동화하는 Flyway 마이그레이션 도구를 프로젝트에 추가합니다.

### 왜 필요한가?
- **현재 문제:** ShedLock 테이블을 로컬과 배포 서버에 **수동으로 생성**해야 함
- **해결:**
  - 마이그레이션 스크립트를 코드에 포함
  - 서버 시작 시 자동으로 테이블 생성
  - 로컬/개발/운영 환경 모두 자동 적용
- **장점:**
  - DB 변경 이력 관리
  - 버전 관리 가능
  - 팀원 간 DB 스키마 동기화

### 변경 파일

#### 1. build.gradle - Flyway 의존성 추가
```gradle
// Flyway (DB 마이그레이션)
implementation 'org.flywaydb:flyway-core'
implementation 'org.flywaydb:flyway-mysql'
```

#### 2. application-dev.yml - Flyway 설정
```yaml
flyway:
  enabled: true                          # Flyway 활성화
  baseline-on-migrate: true              # 기존 DB에도 적용 가능
  locations: classpath:db/migration      # 마이그레이션 스크립트 위치
```

#### 3. 마이그레이션 스크립트 생성

**파일:** `src/main/resources/db/migration/V1__Create_shedlock_table.sql`

```sql
-- ShedLock 테이블 생성
-- 블루그린 배포 시 스케줄러 중복 실행 방지용

CREATE TABLE IF NOT EXISTS shedlock (
    name VARCHAR(64) NOT NULL COMMENT '스케줄러 작업 이름',
    lock_until TIMESTAMP(3) NOT NULL COMMENT '락 만료 시간',
    locked_at TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '락 획득 시간',
    locked_by VARCHAR(255) NOT NULL COMMENT '락을 획득한 인스턴스 식별자',
    PRIMARY KEY (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='스케줄러 분산 락 테이블';
```

### 작동 방식

1. 서버 시작 시 Flyway가 `flyway_schema_history` 테이블 확인
2. 실행되지 않은 마이그레이션 스크립트 검색
3. 버전 순서대로 자동 실행
4. 실행 이력을 `flyway_schema_history`에 기록

### 마이그레이션 파일 명명 규칙

```
V{버전}__{설명}.sql

예시:
V1__Create_shedlock_table.sql
V2__Add_vote_index.sql
V3__Alter_member_table.sql
```

- `V`: 버전 마이그레이션 (필수)
- `{버전}`: 숫자 (1, 2, 3... 또는 1.0, 1.1 등)
- `__`: 언더스코어 2개 (필수)
- `{설명}`: 마이그레이션 설명 (언더스코어로 단어 연결)

### 예상 효과
- ✅ 로컬 테스트 시 자동으로 테이블 생성
- ✅ 배포 시 자동으로 운영 DB에 테이블 생성
- ✅ 팀원들의 로컬 DB 자동 동기화
- ✅ DB 스키마 변경 이력 추적 가능

### 추가 비용
**없음**

### 주의사항
- 이미 실행된 마이그레이션은 수정하지 말 것 (새 버전으로 추가)
- `baseline-on-migrate: true` 덕분에 기존 DB에도 적용 가능
- 첫 실행 시 `flyway_schema_history` 테이블이 자동 생성됨
